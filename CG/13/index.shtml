Exercise 13: Line Rasterization

Topic: DDA and Bresenham Lines (20 points)

In the previous software rasterization tasks, triangles were drawn by testing every pixel in the image. That was intentionally simple, but it is not how efficient rasterizers work. This exercise begins the transition to more efficient rasterization with the simplest primitive: a line segment.

The application already creates an image, draws a background grid, and calls two line drawing functions for a set of test lines. Your task is to implement these functions. The left half of the reference image uses DDA; the right half uses Bresenham.

Exercise 1: DDA Line Rasterization (8 points)

Implement drawLineDDA in main.cpp. DDA treats the line as a continuous segment. The number of steps is determined by the dominant axis. The current position is then advanced by a floating-point increment and rounded to the nearest pixel.

Start with the difference between the endpoints: dx = b.x - a.x and dy = b.y - a.y. The line can then be written as sampled positions: p(i) = a + i * (dx / steps, dy / steps). The key choice is steps = max(abs(dx), abs(dy)). This advances the dominant axis by exactly one pixel per iteration. For a mostly horizontal line this is a step in x; for a mostly vertical line it is a step in y. The other coordinate changes by a fraction each time.

Since the sampled position usually does not lie exactly on integer pixel coordinates, DDA rounds the current floating-point position before calling setPixel. This makes the algorithm easy to understand and naturally handles all slopes, but it requires floating-point additions and a rounding operation for every drawn pixel.

  1. Compute dx, dy, and steps = max(abs(dx), abs(dy)).
  2. Handle the case where both endpoints are the same pixel.
  3. Compute xStep and yStep.
  4. Start at endpoint a.
  5. Draw steps + 1 rounded positions with setPixel, including both endpoints.
  6. After each pixel, add xStep and yStep to the floating-point position.

Exercise 2: Bresenham Line Rasterization (10 points)

Implement drawLineBresenham. Bresenham avoids floating-point positions and rounding. Instead, it stores an integer error term that decides when the discrete line path must advance in x, in y, or in both directions.

The variant used here works for all directions and slopes. First compute the positive distances dx = abs(b.x - a.x) and dy = abs(b.y - a.y). Then determine the signs sx and sy. They indicate whether a step in x or y moves toward the endpoint. With these signs, the same loop can draw lines left-to-right, right-to-left, upward, downward, shallow, steep, and diagonal.

The error value tracks the accumulated discrete deviation from the ideal mathematical line and decides which neighboring pixel is chosen next. Initialize it with error = dx - dy. In each iteration, draw the current pixel and then compute doubledError = 2 * error. Doubling allows comparisons against half-pixel decision boundaries without using fractions.

If doubledError > -dy, enough error has accumulated to take a step in x; then subtract dy from the error. If doubledError < dx, take a step in y; then add dx to the error. Sometimes both conditions are true, producing a diagonal step. This compact Bresenham variant can handle all octants with the same loop.

  1. Use absolute values for dx and dy.
  2. Use the signs sx and sy so the algorithm works in all directions.
  3. Initialize the error with dx - dy.
  4. Draw the current pixel before changing the current position.
  5. Stop the loop when the current point reaches endpoint b.
  6. Use 2 * error to decide whether to step in x, in y, or in both directions.
  7. Update the error on each step so the next iteration knows how far the discrete path has drifted.

Exercise 3: Compare the Algorithms (2 points)

Run the program and compare both halves of the image. Both methods should draw visually similar lines, but they use different computations. Briefly explain why DDA is easier to understand and why Bresenham is more attractive for an integer rasterizer.
