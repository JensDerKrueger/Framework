Acceleration Structure Idea

The growth simulation repeatedly creates new particles and moves them randomly until they are close enough to an existing particle. Without a spatial data structure, every new position would have to be tested against all particles that have already been inserted. The octree reduces this search by recursively subdividing space into eight subregions.

Each OctreeNode describes an axis-aligned box between minPos and maxPos. As long as a node is a leaf, it stores its particles directly in elements. Once a leaf contains too many elements and the maximum depth has not been reached, the node is split into eight children.

Center and Child Index

computeCenter computes the center of the node box. This center splits the space into two halves along the x, y, and z axes, producing eight octants.

subtreeIndex encodes the matching octant as a number from 0 to 7. Points left of the center contribute 0 in x direction, points to the right contribute 1. For y, either 0 or 2 is added, and for z, either 0 or 4. The sum is the child index.

This encoding matters because it uses the same ordering of the eight children everywhere: during insertion, during splitting, and later during the distance search.

Inserting Into the Octree

OctreeNode::add first distinguishes between leaf nodes and inner nodes. If the current node is a leaf, the new particle is inserted directly into elements. The solution then checks whether the allowed element count has been exceeded and whether further subdivision is still allowed.

If both conditions are true, the node calls split. It also resets warranty to 20. This variable belongs to the visualization: freshly split nodes are drawn more prominently for a few frames and then fade out.

If the current node is already an inner node, nothing is stored locally. Instead, subtreeIndex determines the correct child, and insertion is recursively forwarded to that child. During this recursive call, maxDepth is reduced by one.

Why the Depth Is Counted Down

Here, maxDepth represents the remaining recursion depth. The root node starts with the value stored in Octree. With every descent into a child, this value becomes smaller. A leaf may only be split if maxDepth > 0.

This creates a hard upper bound for the tree depth. That is important because particles can be located very close to each other. Without a depth limit, such cases could lead to ever smaller cells and uncontrolled recursion.

Creating Eight Child Nodes

split first computes the center of the current box. It then creates eight new OctreeNode objects. Each child receives its own sub-box, whose bounds are composed from minPos, center, and maxPos.

The first four children lie in the lower z half, and children 4 through 7 lie in the upper z half. Within each z half, the children are again distinguished by y and x. This order exactly matches the index computation in subtreeIndex.

Redistributing Existing Elements

After the children have been created, the inner node may no longer keep its particles locally. Therefore, the solution iterates over all previously stored elements, determines the matching child index for each element, and calls add on that child.

This recursive call deliberately uses the same path as inserting a new particle. If a child becomes too full because of the redistributed elements, it can immediately split again as long as the remaining depth allows it. Only after that is the old node's local element list cleared.

Interaction With the Distance Search

The actual nearest-neighbor search was already present in the framework. However, it assumes that elements are stored correctly in leaves and that inner nodes contain only children. This is exactly the invariant established by add and split.

minSqDistApprox first searches the child that contains the query point. If that child is an empty leaf, all children are considered as a fallback. minSqDist then refines the search: a child is only visited if its box can intersect the current search sphere. This skips many subtrees.

Why This Is Enough for the Growth Simulation

For the simulation, each test point only needs the distance to the nearest existing particle. If this distance is small enough, the new particle sticks to the structure and is inserted into the tree with Octree::add. Thanks to spatial subdivision, the computational cost grows much more slowly than with a comparison against all previous particles.

The octree visualization is not an additional part of the exercise, but it helps with validation: as the particle structure becomes denser, new subdivisions should appear where many particles occupy small spatial regions.
